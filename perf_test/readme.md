# СУБД ДЗ 4. Оптимизация работы СУБД 

## 1) Выбор сущности

Основной сущностью для тестирования были выбраны **плейлисты.**
Так как из сущностей, имеющих в нашем проекте отдельный CRUD, плейлисты наиболее часто используются пользователем. 
К тому же у них простой и понятный endpoint создания и чтения. 
Из других кандидатов были рассмотрены комментарии, но они требуют дополнительного усложнения тестирования из-за использования WebSocket.

---

## 2) Что тестируем

### Endpoint создания (Write)
`POST /api/v1/playlists`  
Задача: создать 100 000 сущностей playlist.

### Endpoint чтения (Read)
`GET /api/v1/playlists/my?limit=10&offset=...`  
Задача: проверить чтение на получившейся базе и рассмотреть влияние конкуренции и OFFSET.

---

## 3) Инструмент нагрузочного тестирования

В качестве инструмента был выбран **wrk**, благодаря наличию консольного режима, низкому порогу входа и возможностью писать Lua-скрипты.

## 6) Выводы по работе

В рамках работы удалось создать около 123 тысяч плейлистов через API с использованием нагрузочного тестирования. При последовательной записи (1 поток, 1 соединение) сервис показал удовлетворительное поведение: средняя пропускная способность составила примерно 92–97 запросов в секунду, средняя задержка находилась в диапазоне 10–11 мс, а p99 - около 23–25 мс. Это говорит о том, что операция записи основной сущности работает предсказуемо при умеренной нагрузке.

Чтение данных с единственного клиента также выполняется достаточно быстро. Медианная задержка составила около 11 мс, а 99-й перцентиль - порядка 33 мс. Для одиночного пользователя или низкой конкуренции считаем результат удовлетворительным.

При увеличении конкуренции латентность резко возрастает: p50 достигает примерно 135 мс, p99 - около 240 мс. При этом пропускная способность почти не увеличивается по сравнению с одиночным сценарием. Это указывает на наличие узкого места, вероятнее всего связанного с базой данных или пулом соединений.

Отдельно стоит отметить нагрузку при большом offset: наблюдается рост задержек и снижение RPS по сравнению с запросами без большого смещения. Т.е. база сканирует и отбрасывает большое количество строк перед тем, как вернуть нужный диапазон.

Итого, базовая реализация удовлетворительно работает на объёме в 100k сущностей, но при росте конкуренции и использовании offset возникают проблемы масштабирования. 

## 7) Что можно улучшить?

### Анализ узких мест и возможные улучшения

По результатам тестирования можно выделить несколько узких мест.

Во-первых, отсутствует составной индекс `(user_id, created_at DESC)`. 
В результате база либо использует индекс по дате с последующей фильтрацией по пользователю, либо индекс по пользователю с дополнительной сортировкой. 
На небольших данных это незначительно, но при их росте приводит к значительному снижению производительности. 

Во-вторых, сам подход с `LIMIT/OFFSET` плохо масштабируется при больших объемах данных. 
Возможно, стоит рассмотреть подход с keyset pagination.

В-третьих, результаты тестов чтения под конкуренцией указывают на узкое место в пуле соединений к базе данных.
Вероятно, стоит увеличить количество открытых соединений к БД и согласовать размер пула с количеством воркеров.
